# 会话同步记录（node.txt）

更新时间：2026-03-05

## 一、项目目标
- 当前优先做 v1：线程池并发文件服务器。
- 后续再做 v2：epoll 版本。
- 功能：`list/get/put/quit`，支持多客户端并发，服务端打印客户端访问时间。

## 二、部署与编译结论
- 服务端运行在板子（AArch64），客户端在虚拟机（x86_64/Linux）。
- 两端编译器不同没问题，关键是协议一致。
- 网络传输必须做字节序转换：`htonl/htons/ntohl/ntohs`。

## 三、协议与类型约定
- 协议头 `msg_header_t`：`magic/cmd/status/length`。
- `length` 建议统一 `uint32_t`（避免 `size_t` 在 32/64 位差异）。
- `send_all/recv_all` 返回值使用 `ssize_t`。
- `__attribute__((packed))` 在 GCC/Clang 可用；Windows 本地可能仅是静态检查误报。

## 四、当前 v1_threadpool 结构
- 目录已搭好，`include/*.h` 基本齐全。
- `src/` 包含：`main.c server.c threadpool.c protocol.c file_ops.c log.c`。
- 目前开发方式：用户手敲代码，Copilot 提供分文件模板与解释。

## 五、已确认的关键修正点
- `protocol_io.h` 的 include guard 需独立（不能和 `protocol.h` 一样）。
- `protocol.c` 使用可靠收发循环，并处理 `EINTR`。
- `send_header/recv_header` 已按网络字节序思路实现。

## 六、已给出的 threadpool.c 模板要点
- 线程池结构：线程数组 + 任务队列 + 互斥锁 + 条件变量 + stop 标志。
- API：`thread_pool_create / thread_pool_submit / thread_pool_destroy`。
- worker 流程：等待任务 -> 出队 -> 调 handler -> 释放节点。
- 重点：handler 中要确保 `close(client_fd)`，避免 fd 泄漏。

- 若偶发失败：切 VPN 节点，或改 SSH 远程推送。

## 十二、今晚新增进展（2026-03-05，补充）

### 1) AArch64 交叉编译报错已定位并修复
- 现象：`make server_aarch64` 时调用了 `aarch64-none-linux-gnu-cc`，报“没有那个文件或目录”。
- 根因：Make 内置默认 `CC=cc`，原写法 `SERVER_CC = $(CROSS_COMPILE)$(CC)` 被拼成 `...-cc`。
- 处理：`v1_threadpool/Makefile` 已改为显式交叉编译器变量：
	- `AARCH64_CC ?= $(CROSS_COMPILE)gcc`
	- AArch64 的编译/链接规则统一使用 `$(AARCH64_CC)`。
- 结论：避免了 `CC=cc` 污染，交叉编译行为稳定。
